## 4.1 📋 存储器概述

### 4.1.1 存储器分类

#### 🌲 按存储介质（物理材料）分

| 类型 | 原理 🛠️ | 特点 | 例子 |
|:----|:---------|:-----|:-----|
| **半导体存储器** 🔬 | 用晶体管/电容存储电荷 | 速度快、功耗低、体积小 | 内存条、CPU缓存、U盘 |
| **磁介质存储器** 🧲 | 用磁性材料剩磁方向存储 | 容量大、速度慢、体积大 | 机械硬盘(HDD)、磁带 |
| **激光存储器** 💿 | 用激光在光盘烧出凹坑 | 便携、廉价、易保存 | CD/DVD/蓝光 |

#### 🚶 按存取方式

| 类型 | 类比 | 读写方式 | 例子 |
|:----|:----|:---------|:-----|
| **随机存取(RAM)** 🎲 | 课表"第3节课"直接翻到 | 访问任意地址时间相同 | 内存、寄存器 |
| **顺序存取(SAM)** 🚶 | 磁带从头放 | 只能从前往后挨个访问 | 磁带 |
| **直接存取(DAM)** 🚗 | CD唱片跳到第5首歌 | 先快速定位再顺序读写 | 机械硬盘、光盘 |

#### ✏️ 按读写功能

| 类型 | 能否改写 | 用途 |
|:----|:--------|:-----|
| **只读存储器(ROM)** 🔒 | 出厂固定，不能（或很难）改写 | BIOS/UEFI固件 |
| **读/写存储器(RAM)** ✏️ | 可以读也可以写 | 内存、缓存 |

#### 🔥 按断电后信息是否保留

| 类型 | 断电后... | 英文名 | 例子 |
|:----|:---------|:------|:-----|
| **易失性** 🔥 | 立刻丢失！ | Volatile | DRAM(内存条)、SRAM(Cache) |
| **非易失性** 🛡️ | 信息依然保存 | Non-Volatile | ROM、Flash(U盘/SSD)、磁盘 |

> **💡 记忆技巧：** 内存条一拔数据就没了（易失），U盘拔了插别的电脑还有（非易失）

#### 🖥️ 按在系统中的作用

```
CPU内部: 寄存器(最快最小) → Cache(L1/L2/L3)
CPU外部: 主存(DRAM) → 辅存(硬盘/U盘/光盘)
```

### 4.1.2 存储器技术指标 📏

#### 三个核心指标

| 指标 | 含义 | 类比 | 典型值 |
|:----|:-----|:----|:------|
| **存储时间** ⏱️ | 从CPU说"我要读"到数据出来的时间 | 你喊我名字到我答"到" | SRAM ~1ns, DRAM ~50ns, 硬盘 ~10ms |
| **存储周期** 🔄 | 连续两次访问的最小间隔 | 上课最少要等45分钟才能下课再上课 | 存储时间+恢复时间 |
| **存储器带宽** 🌊 | 单位时间内能传多少数据 | 高速公路的车流量 | Byte/s 或 bit/s |

> **⚠️ 注意：** 存储周期 ≥ 存储时间（读完需要时间恢复电路状态）

**带宽计算公式：**
```
带宽 = (每个存储周期传输的数据量) / 存储周期
```

### 4.1.3 存储系统层次结构 🏗️

> **核心理念：** 用"分级存储"来平衡**速度、容量和价格**三者关系

```
          ⬆ 速度越来越快 ⬆           ⬇ 容量越来越大 ⬇
          ⬆ 价格越来越贵 ⬆           ⬇ 价格越来越便宜 ⬇

       寄存器  (<1ns, 几百B)       ← CPU内部
          ↓
  ┌─ L1 Cache  (~1ns, 32KB)    ← CPU内部 (最快缓存)
  │  L2 Cache  (~10ns, 256KB)  ← CPU封装内
  │  L3 Cache  (~20ns, 8MB)    ← CPU封装内 (最慢缓存)
  └─                           ← 速度递减
          ↓
       主存(DRAM) (~50ns, 8-32GB)  ← 内存条
          ↓
       固态盘SSD (~0.1ms, 512GB)   ← 外存 (其实内部也是半导体)
          ↓
       机械硬盘HDD (~10ms, 2TB+)   ← 外存
```

> 🔑 **关键规律：** 越往上**速度越快、容量越小、价格越贵**；越往下越**速度越慢、容量越大、价格越便宜**

### 4.1.4 主存的基本结构

```
地址总线 ──→ MAR (地址寄存器) ──→ 存储体 ──→ MDR (数据寄存器) ──→ 数据总线
                   控制信号(R/W#, CS#) ──→ 控制电路
```

| 部件 | 全称 | 作用 |
|:----|:----|:-----|
| **MAR** | Memory Address Register | 存储要访问的内存地址 |
| **MDR** | Memory Data Register | 暂存读出的数据或要写入的数据 |
| **存储体** | Memory Bank | 由成千上万个存储单元组成的阵列 |

### 4.1.5 主存中数据的存放

#### 字节编址

现代计算机都是**按字节编址**的，每个字节都有一个唯一的地址。

```
一个 int 占 4 个字节，占用连续 4 个地址
地址: 0x1000 | 0x1001 | 0x1002 | 0x1003
数据:  [低字节]   [  ]    [  ]    [高字节]
```

#### 大小端存储 ⭐

以存储 `0x12345678` 到地址 `0x1000` 为例：

| 方式 | 0x1000 | 0x1001 | 0x1002 | 0x1003 | 常见架构 |
|:----|:------:|:------:|:------:|:------:|:---------|
| **大端序** 🔛 | 12 | 34 | 56 | 78 | 网络协议、PowerPC |
| **小端序** 🔚 | 78 | 56 | 34 | 12 | **x86、x64** |

> 💡 **记忆口诀：** 小端 = 低位字节存低地址（"小"和"低"配对）

#### 数据对齐 ⭐

> CPU要求数据存放在其大小的整数倍地址上

```c
// 假设有以下变量
int i;      // 4字节 → 地址需是4的倍数
short k;    // 2字节 → 地址需是2的倍数
double x;   // 8字节 → 地址需是8的倍数（64位机）
char c;     // 1字节 → 任何地址都行
```

```
对齐存放（推荐）✅：
地址: 0x00 int i | 0x04 short k | 0x06 (空2B) | 0x08 double x ...

未对齐存放 ❌：
double x 可能跨在 0x06~0x0D，CPU需要访问两次才能读完整！

结论：✅ 对齐 = 速度快 | ❌ 不对齐 = 节省空间但慢
```

## 4.2 🔬 半导体存储器

### 4.2.1 SRAM（静态随机存储器）

> **一句话：** 速度极快但很贵的"跑车🏎️"，用来做**CPU缓存**

#### 🏗️ SRAM存储单元 — 六管结构

每个SRAM存储单元由 **6个MOS管** 组成：

```
                   VDD(电源)
                    │
       ┌──── T3 ───┴─── T4 ────┐
       │         │             │
  位线B ──┤ T5  T1 ═══ T2  T6 ├── 位线B反
       │         │             │
       └──── T7 ────┬─── T8 ──┘
                    │
                   GND
```

**用"跷跷板"来理解 🎢**

想象一个跷跷板：
- **T1导通、T2截止** → 跷跷板停在左边 → 存"0"
- **T1截止、T2导通** → 跷跷板停在右边 → 存"1"
- 这两个状态是**稳定**的（一旦确定就不需要外力维持）
- T3、T4（负载管）：就像跷跷板的润滑油，确保状态不自己变化

**读操作：**
```
① X地址译码 → T5、T6导通 → A点连到位线
② Y地址译码 → T7、T8导通 → A点电位输出到I/O端
③ 读出A点高低电平 → 得到存储的数据
```

**写操作：**
```
① I/O端输入要写入的数据(0或1)
② 通过T7/T8 → 位线 → T5/T6
③ 强行改变T1/T2的状态
```

**SRAM的缺点：❌**
- 1个bit需要**6个MOS管**→ 占用面积大 → 集成度低
- 功耗大 → 发热严重
- 单位容量成本高 → 贵！

### 4.2.2 DRAM（动态随机存储器）

> **一句话：** 容量大、价格便宜但需要"充电"的家用车🚗，用来做**内存条**

#### SRAM → DRAM的改进

```
SRAM (6管)                    DRAM (4管)
  T3 T4 (负载管)                 →   去掉!   → 更小了
  T1 T2 (工作管)                 →   保留
  T5 T6 T7 T8 (门控管)          →   保留
                                 →   增加电容C1 C2

  DRAM (1管)  ← 最终进化
  1个MOS管 + 1个电容 → 集成度大大提高！
```

> **DRAM的原理：** 电容有电荷 = 存"1"；电容没电荷 = 存"0"

#### 单管DRAM存储单元 ⭐

```
字线(行选通)
    │
    ├── T1 (开关管)
    │     │
   位线    └── C (存储电容，关键！)

每个bit = 1个晶体管 + 1个电容
```

#### DRAM读写流程（单管）

| 步骤 | 操作 | 说明 |
|:----|:-----|:------|
| ① **预充** 🔋 | 给位线充电到 Vcc/2 | 做好准备 |
| ② **行选通** 🚪 | T1导通，电容C连上位线 | 电荷开始"分享" |
| ③ **检测放大** 🔍 | 读出放大器检测微小电压差 | 就像用天平称出蚊子重量！ |
| ④ **数据恢复** 🔄 | 如果是1，给电容重新充满电 | 因为读出是**破坏性**的！ |
| ⑤ **列选通** 🚪 | 将数据输出到I/O引脚 | |

> **⚡ 大招理解：** DRAM读出数据的过程会把电容里的电荷放掉，所以读出后必须**恢复**（重新充电），这叫"破坏性读出"

#### DRAM为什么要刷新？⏰

```
电容就像一个有洞的水桶🚰：
- 刚装满水（存1）→ 水慢慢漏掉 → 过一会儿就没水了（变成0了！）
- 必须在漏完之前重新加水 → 这就是"刷新"
```

#### DRAM刷新的三种方式 ⭐

```
① 集中刷新 🎯
   读写/维持    集中刷新
   ████████████████████│██████│
   正常使用 2ms       死区(不能读写)
   
   优点：控制简单
   缺点：有一段"死区"不能访问内存

② 分散刷新 🔄
   ████│████│████│████│████│
   读写+刷新  读写+刷新  读写+刷新
   每个存取周期都分一半给刷新
   
   优点：无死区
   缺点：刷新次数太多(2ms要做20000次刷新)

③ 异步刷新 ⏱️ ⭐ (最常用)
   ████│████│████│████│████│
   操作 刷新一行 操作 刷新一行 ...
   ←每15.6μs刷一行→
   
   优点：无死区、效率高
   128行 × 15.6μs ≈ 2ms
```

#### SRAM vs DRAM 终极对比 ⭐

| 对比项 | SRAM | DRAM |
|:------|:----:|:----:|
| **存储原理** 🧬 | 触发器（6个MOS管） | 电容（1MOS+1电容） |
| **速度** 🏃 | **快** (1~2ns) | 慢 (50ns) |
| **集成度** 🔢 | **低**（1个bit占6个管面积） | 高（1个bit才1管+1电容） |
| **功耗** 💡 | 高（6个管一直在耗电） | **低** |
| **价格** 💰 | 贵 | **便宜** |
| **需要刷新？** | ❌ 不需要 | ✅ **每2ms必须刷新一次** |
| **用途** 🎯 | **CPU缓存** | **主存（内存条）** |

### 4.2.3 只读存储器 (ROM)

#### ROM家族图谱

```
        ROM (掩膜ROM)
        │ 出厂时用光刻掩膜固定，不能修改
        ↓
       PROM (可编程ROM)
        │ 用户用编程器烧断熔丝，只能写一次
        ↓
       EPROM (可擦除可编程ROM)
        │ 紫外线照射擦除，可以反复写
        ↓
      EEPROM (电可擦除ROM)
        │ 电信号擦除，不用紫外线灯了，方便！
        ↓
   ⭐ Flash (闪存)
        │ 本质上属于EEPROM，但擦写速度更快、集成度更高
        │ NOR Flash: 代码可直接运行(XIP)
        │ NAND Flash: 容量大、速度快 → U盘、SSD
```

| 类型 | 编程次数 | 擦除方式 | 典型用途 |
|:----|:--------|:---------|:---------|
| **ROM** | 1次（出厂固定） | ❌不能擦除 | 早期固件 |
| **PROM** | 1次（熔丝） | ❌不能擦除 | 少量生产 |
| **EPROM** | 多次 | ☀️紫外线擦（20分钟） | 开发调试 |
| **EEPROM** | 多次 | ⚡电擦除（字节级） | 参数存储 |
| **Flash** 🚀 | 多次 | ⚡电擦除（块级） | **U盘、SSD、BIOS** |

#### 🇨🇳 长江存储与3D NAND

```
2018: 32层 3D NAND
2020: 128层 3D NAND (QLC 1.33Tb) 全球首款！
2022: 232层 3D NAND 研发成功
2024: 致态TiPro9000 (PCIe 5.0, 读14.5GB/s)
      192层量产，存储密度20Gb/mm²
      
Xtacking 架构：把存储阵列和控制电路"叠"在一起
            → 密度高、速度快
```

### 4.2.4 DRAM的发展史

```
FPM DRAM ─→ EDO DRAM ─→ SDRAM ─→ DDR SDRAM ─→ DDR2 ─→ DDR3 ─→ DDR4 ─→ DDR5
                                 │
                           每时钟传2次数据
                           (上升沿+下降沿)
```

> 💡 DDR = **D**ouble **D**ata **R**ate，一个时钟周期传两次数据，比SDRAM快一倍

## 4.3 🔗 主存的组织及与CPU的连接

### 基础连线

```
CPU                       存储器芯片
┌─────────┐              ┌───────────┐
│ A0~An   │──地址总线──→│ A (Address)│
│ D0~Dm   │←数据总线──→│ D (Data)   │
│ MREQ#   │──访存请求──→│ CS#  (片选)│
│ R/W#    │──读写控制──→│ WE#  (写使能)│
└─────────┘              └───────────┘
```

**引脚功能解释：**
- **A (Address)**：地址线，输入要访问的地址
- **D (Data)**：数据线，读写数据传输
- **CS# (Chip Select)**：片选，只有选中了芯片才会工作（#表示低电平有效）
- **WE# (Write Enable)**：写使能（#表示低电平时写入，高电平时读出）

### 存储器扩展 ⭐

#### ① 字长扩展（位扩展）

> **场景：** 芯片的**数据位宽**不够CPU的数据总线宽度

**例子：** 用 `256K×1` 的芯片组成 `256K×8` 的存储器

```
CPU D7 D6 D5 D4 D3 D2 D1 D0
     │  │  │  │  │  │  │  │
     ├──┤  ├──┤  ├──┤  ├──┤
     │片0│  │片1│  │片2│  │片3│  ← 每个芯片贡献1位
     │D0 │  │D1 │  │D2 │  │D3 │
     └──┘  └──┘  └──┘  └──┘
     片4  片5  片6  片7
     D4   D5   D6   D7
     
     所有芯片的地址线(A)连在一起 → 并联
     所有芯片的控制线(CS, WE)连在一起 → 并联
     数据线各自独立 → 各贡献1位
```

**公式：**
```
需要芯片数 = CPU数据位宽 / 芯片位宽 = 8 / 1 = 8片
```

> **关键理解：** 8个芯片**同时工作**，每个芯片管1位数据，拼成一个完整的字节

#### ② 字数扩展（地址扩展）

> **场景：** 芯片的**容量**不够CPU需要的总容量

**例子：** 用 `256K×8` 的芯片组成 `1M×8` 的存储器

```
高位地址线 A19 A18 → 3:8译码器
                        │
      ┌────┬────┬────┬────┐
     Y0   Y1   Y2   Y3
      │    │    │    │
    片0  片1  片2  片3
 (256K) (256K) (256K) (256K)
 
所有芯片的数据线(D)连在一起
所有芯片的低18位地址线(A17~A0)连在一起
```

**地址范围：**
```
片0: 0x00000 ~ 0x3FFFF
片1: 0x40000 ~ 0x7FFFF
片2: 0x80000 ~ 0xBFFFF
片3: 0xC0000 ~ 0xFFFFF
```

> **关键理解：** 同一时刻只有**1个芯片工作**，由高位地址译码决定谁"值班"

#### ③ 综合扩展 ⭐

> **场景：** 芯片位数不够**且**容量也不够

**公式：**
```
需要的芯片总数 = (M/l) × (N/k)
其中：
  目标存储器 = M × N 位
  单芯片    = l × k 位
```

**例子：** 用 `256K×1` → 组成 `1M×8`
```
字数方向 = 1M/256K = 4
位方向   = 8/1 = 8
总芯片数 = 4 × 8 = 32片！
```

### 综合电路例题 ⭐⭐⭐

> CPU地址线A15~A0，数据线D7~D0
> 存储器分配：
> - ROM: 0x0000 ~ 0x3FFF (16K)
> - 保留: 0x4000 ~ 0x5FFF (8K)
> - RAM: 0x6000 ~ 0xFFFF (40K)

**解题步骤：**
1. **写地址范围（二进制）**
   ```
   0x0000 ~ 0x3FFF: A15 A14 A13 = 000
   0x4000 ~ 0x5FFF: A15 A14 A13 = 001
   0x6000 ~ 0xFFFF: A15 A14 A13 ≥ 011
   ```

2. **确定需要的芯片**
   - ROM 16KB → 用 1片 16K×8 或 2片 8K×8
   - RAM 40KB → 用 10片 4K×8 （因为40K/4K=10）

3. **设计译码逻辑**
   ```
   A15 A14 A13 → 3:8译码器
   000 → Y0 → ROM片选
   001 → Y1 → 保留(不接)
   011~111 → Y3~Y7 → RAM片选
   ```

4. **连线**
   地址线、数据线、控制线分别连接
   片选通过译码器输出

## 4.4 ⚡ 并行主存系统

### 为什么要并行？

> CPU太快了，内存太慢了！瓶颈在内存上。
> 
> 解决方案一：把内存变得更宽（一次多读几个字）
> 解决方案二：让多个内存模块流水工作

### 双端口存储器

```
双向同时存取
┌─────────────────────────────────┐
│           双端口RAM              │
│ 左端口 ├───── 存储体 ─────┤ 右端口 │
│  (CPU1)                      (CPU2) │
└─────────────────────────────────┘

✅ 左右端口访问不同地址 → 并行工作，速度翻倍！
❌ 左右端口访问相同地址 → 冲突！由BUSY信号裁决
```

### 多模块存储器

#### ① 单体多字

> 多个模块**完全同步**，同时读写

```
地址寄存器AR (共用)
    │
 ┌──┬──┬──┬──┐
 │M0│M1│M2│M3│  ← 4个模块同时工作
 └──┴──┴──┴──┘
    │
 多字长数据总线 → 一次传4个字

性能：一个周期读4个字，相当于4倍带宽！
```

#### ② 多体多字（双通道内存）

> 多个模块**异步**工作，各自独立

```
两个独立通道：
  内存控制器1 ←→ 内存条A1, A2  
  内存控制器2 ←→ 内存条B1, B2

Unganged模式：两条内存条独立编址，灵活调度
Ganged模式：  两条内存条合成一条更宽的总线
```

#### ③ 多体交叉存储器 ⭐⭐ (重点！)

> **核心理念：** 把地址交错的分配到不同的模块，让模块像流水线一样工作

**编址方式对比：**

```
顺序编址（高位交叉）：
  模块0: 地址 0~7
  模块1: 地址 8~15
  模块2: 地址 16~23
  模块3: 地址 24~31
  → 一次只能用一个模块，性能无提升

交叉编址（低位交叉）⭐：
  模块0: 地址 0, 4, 8, 12, ...
  模块1: 地址 1, 5, 9, 13, ...
  模块2: 地址 2, 6, 10, 14, ...
  模块3: 地址 3, 7, 11, 15, ...
  → 连续地址被分配到不同模块，可以流水工作！
```

**流水工作示意图：**
```
模块0: [读0]╌╌╌╌╌╌╌╌[读4]╌╌╌╌╌╌╌╌[读8]╌╌╌╌╌╌╌╌
模块1: ╌╌[读1]╌╌╌╌╌╌╌╌[读5]╌╌╌╌╌╌╌╌[读9]╌╌╌╌╌╌
模块2: ╌╌╌╌[读2]╌╌╌╌╌╌╌╌[读6]╌╌╌╌╌╌╌╌[读10]╌╌╌
模块3: ╌╌╌╌╌╌[读3]╌╌╌╌╌╌╌╌[读7]╌╌╌╌╌╌╌╌[读11]╌
        ←T→←τ→←τ→←τ→         ← 时间 →
        |←一个存储周期→|
```

**连续读取n个字的时间：**

| 方式 | 计算公式 | 例子（T=50ns, τ=10ns, n=10） |
|:----|:---------|:----------------------------|
| **交叉存取** 🚀 | `t₁ = T + (n-1)τ` | 50 + 9×10 = **140ns** |
| **顺序存取** 🐢 | `t₂ = nT` | 10×50 = **500ns** |

**交叉存取度：** `m = T / τ`

> **💡 通俗理解：** 
> 顺序存取就像只有一个收银台，顾客排队一个一个结账
> 交叉存取就像开了多个收银台，虽然每个人结账时间不变，但总体速度快多了！

## 4.5 🚀 高速缓冲存储器 (Cache) ⭐⭐⭐

### 为什么需要Cache？

```
CPU速度 (~1ns)      ← 博尔特 (9.58秒/100米)
    │ 差 50 倍！
DRAM速度 (~50ns)    ← 普通人跑100米
```

解决思路：在CPU和主存之间加一个容量适中但速度快的SRAM → **Cache**

### 4.5.1 缓存思想

#### 🏫 自习室的凳子和书包

```
你每次学习要从书包里拿书
- 每次弯腰翻书包 → 很慢
- 把常用书放在桌面上 → 随手就能拿到

Cache = 桌面（快，但地方小）
主存 = 书包（大，但翻起来慢）
```

### 4.5.2 程序局部性 ⭐

#### ⏰ 时间局部性

> **刚用过的东西，很可能马上又用**

```c
for (i = 0; i < 1000; i++) {
    sum += x;      // x 反复使用 → 时间局部性好
}
```

> **就像：** 你写作业时，最常用的笔就握在手里，不会每次都去笔袋里翻

#### 🌍 空间局部性

> **用了一个东西，它附近的东西也快用了**

```c
int a[1000];
for (i = 0; i < 1000; i++) {
    sum += a[i];   // 连续访问 a[0], a[1], a[2]... → 空间局部性好
}
```

> **就像：** 你翻到书的第50页，很可能接着看51页、52页

#### 🆚 好局部性 vs 差局部性（经典考点！）

```c
// ✅ 程序A：局部性好 → 按行遍历
for (i = 0; i < 256; i++)
    for (j = 0; j < 256; j++)
        sum += a[i][j];  
// a[0][0], a[0][1], a[0][2]... → 连续内存

// ❌ 程序B：局部性差 → 按列遍历
for (j = 0; j < 256; j++)
    for (i = 0; i < 256; i++)
        sum += a[i][j];  
// a[0][0], a[1][0], a[2][0]... → 跳着访问
// 每访问一个元素就跳到下一行的同一列 → 跨越大！
```

> **💡 关键：** C/C++ 二维数组是**行优先存储**的
> a[i][j] 和 a[i][j+1] 在内存中相邻
> a[i][j] 和 a[i+1][j] 在内存中相差一行的距离

### 4.5.3 Cache的基本概念

#### Cache术语 ⭐

| 术语 | 英文 | 通俗理解 |
|:----|:----|:---------|
| **命中** 🎯 | Hit | 要的东西就在桌面上 |
| **缺失** ❌ | Miss | 桌面上没有，得去书包拿 |
| **块/行** 📦 | Block/Line | 一次从书包拿到桌面上的"一叠纸" |
| **命中率** 📈 | Hit Rate | 拿10次有9次在桌面上 → 命中率90% |
| **缺失率** 📉 | Miss Rate | 1 - 命中率 |
| **命中时间** ⚡ | Hit Time | 从桌面上拿书的时间 |
| **缺失损失** 💸 | Miss Penalty | 去书包翻书的额外时间（远大于命中时间） |

#### 主存地址的划分

Cache以"块"为单位和主存交换数据

```
主存地址 = |←── 块地址 (Tag) ──→|←── 块内偏移 (Offset) ──→|

块地址：确定数据属于主存中的哪一块
块内偏移：确定数据在块内的哪个位置
```

### 4.5.4 Cache四大关键技术 ⭐⭐⭐

#### 🔑 技术一：数据查找 — 相联存储器

**问题：** CPU给出一个地址，怎么知道它在不在Cache里？

**答案：** 用**相联存储器（Associative Memory）**

> **普通RAM：** 给个地址 → 直接去那个地址取数据
> **相联存储器：** 给个数值(Key) → 在所有存储单元里**同时**查找 → 找到匹配的
>
> 就像查字典时：
> - 普通方式：我知道页码 → 翻到那页
> - 相联方式：我知道拼音 → 在所有页中**同时**查找 → 匹配的页码

```
相联存储器结构：
  Key(主存块号) │ Valid(有效位) │ Value(Cache行号/数据)
  ──────────────┼──────────────┼────────────────────
     001        │      1       │        2
     021        │      1       │        4
     ...        │     ...      │       ...
     091        │      1       │        7
```

#### 🔑 技术二：地址映射 ⭐⭐

> 主存中的某一块数据，可以放到Cache的什么位置？

##### ① 全相联映射 💯

```
Cache行号 = 任意（最自由！）

主存块 → Cache行
块0 → 可以放任何一行
块1 → 可以放任何一行
...

特点：
✅ 最灵活，冲突最少
✅ Cache利用率最高
❌ 查找时要和所有行比较 → 需要n路并发比较器 → 硬件贵
❌ 淘汰算法复杂

适用：小容量Cache、TLB
```

**地址划分：**
```
|←── 块地址(标记) Tag ──→|←── 块内偏移 Offset ──→|
```

##### ② 直接相联映射 📍

```
Cache行号 = 主存块号 mod Cache行数

例：Cache有4行
主存块0 → Cache行0
主存块1 → Cache行1
主存块2 → Cache行2
主存块3 → Cache行3
主存块4 → Cache行0（冲突！因为4 mod 4 = 0）

特点：
✅ 查找最快（直接用行索引定位，1次比较）
✅ 硬件最简单
❌ 一山不容二虎 → 多个主存块抢同一行 → 冲突频繁
❌ Cache利用率低

适用：大容量Cache
```

**地址划分：**
```
|←── 区地址(标记) Tag ──→|←── 行索引 Index ──→|←── 块内偏移 Offset ──→|
```

**理解关键：**
```
把主存分成若干个"区"，每个区的大小等于Cache大小
┌─── 区0 ───┐
│ 块0 → Cache行0 │
│ 块1 → Cache行1 │
│ 块2 → Cache行2 │
└────────────┘
┌─── 区1 ───┐
│ 块4 → Cache行0 │  ← 和区0的块0抢同一行！
│ 块5 → Cache行1 │
└────────────┘
```

##### ③ 组相联映射 🔗 ⭐ (最重要！)

> **全相联 × 直接相联 = 组相联的"爱情结晶" 💕**

```
分组方式：k路组相联 → 每组有k行

第一步：用组索引定位到组（就像直接相联）
第二步：在组内全相联查找（就像全相联）
```

**例子：2路组相联（每组2行）**
```
Cache共8行，分成4组，每组2行

主存块0 → 组0（可在行0或行1）
主存块1 → 组1（可在行2或行3）
主存块4 → 组0（和块0同组，但可以在另一行）
```

**地址划分：**
```
|←── 标记 Tag ──→|←── 组索引 Index ──→|←── 块内偏移 Offset ──→|
```

##### 三种映射方式对比 ⭐

| 特性 | 直接相联 | 组相联(k路) | 全相联 |
|:----|:--------|:-----------|:-------|
| 映射位置 | 唯一固定 | 组固定，行可选 | 任何位置 |
| 需要比较次数 | 1次 | k次 | 全部行数 |
| 硬件成本 💰 | **低** | 中 | 高 |
| 冲突概率 | **高** | 中 | **低** |
| Cache利用率 | 低 | 中 | **高** |
| 典型应用 | 大容量Cache | **L1/L2 Cache** | TLB |

#### 🔑 技术三：替换算法 ⭐

> Cache满了，要放入新数据，把谁"踢出去"？

| 算法 | 策略 | 通俗理解 | 特点 |
|:----|:-----|:---------|:-----|
| **FIFO** ⏳ | 先进先出 | 谁先来的谁走 | 简单，但可能踢走"重要人物" |
| **LFU** 📊 | 用得最少的走 | 谁用得少谁走 | 记录使用次数，硬件成本高 |
| **⭐ LRU** 👑 | 最久没用的走 | 谁最久没碰谁走 | **性能最好！最常用！** |
| **随机** 🎲 | 随机踢一个 | 闭着眼睛踢 | 简单，但性能不稳定 |

**LRU的"自习室"理解 🏫：**
```
自习室的座位坐满了。
新同学要进来，管理员需要赶走一个人。
管理员想："看看谁的座位最久没人坐过了？" 
→ 把那个人赶走，新同学坐他的位置。

因为：坐得越久的同学说明他越不需要这个位置（时间局部性）
```

#### 🔑 技术四：写入策略 ⭐

##### 写命中时

| 策略 | 做法 | 安全性 | 速度 |
|:----|:-----|:------|:----:|
| **写穿(Write Through)** 🕳️ | 写Cache同时**也写主存** | ✅ 数据安全 | 🐢 慢（等主存写完） |
| **写回(Write Back)** 🔙 | **只写Cache**，标记为"脏" | ❌ 可能丢数据 | 🚀 **快**（主存以后再说） |

**脏数据问题：**
```
写回策略：
CPU写入Cache → 标记"脏"位 → 继续工作
              ↓（此时Cache和主存不一致！）
    等到该行被替换时 → 才把脏数据写回主存

如果此时突然断电 → 脏数据丢失！⚠️
```

##### 写缺失时

| 策略 | 做法 |
|:----|:------|
| **写分配(Write Allocate)** 📥 | 先把主存块调入Cache，再写Cache |
| **写不分配(No-Write Allocate)** 🚫 | 直接写主存，不调入Cache |

> **常见组合：**
> - 写回 + 写分配（Write Back + Write Allocate）← 性能好，常用
> - 写穿 + 写不分配（Write Through + No-Write Allocate）

### Cache性能分析 ⭐⭐

#### 性能公式

**命中率 h：**
```
h = Nc / (Nc + Nm)
Nc = Cache命中的次数
Nm = 访问主存的次数
```

**平均访问时间 Ta：**
```
Ta = h × Tc + (1-h) × Tm
Tc = Cache访问时间
Tm = 主存访问时间（包括缺失损失）
```

**访问效率 e：**
```
e = Tc / Ta
```

#### 经典例题 ⭐⭐

**例题1：** 主存4MB，Cache 4KB，块长8个字（32位字长），直接映射

**解：**
```
(1) 地址划分
    主存地址位宽 = log₂(4M) = 22位
    块内偏移 = log₂(8×4) = 5位 (8个字，每字4字节)
    Cache行数 = 4KB / 32B = 128行 → 行索引 = 7位
    标记位 = 22 - 5 - 7 = 10位

    地址：| 标记(10位) | 行索引(7位) | 块内偏移(5位) |

(2) 命中率计算
    0-99号单元共100个字 → 100/8 = 13个块（向上取整）
    每块第一次访问不命中，之后全部命中
    重复10次：
    命中率 = (100×10 - 13) / (100×10) = 987/1000 = 98.7%

(3) 平均访问时间（假设Tc=2ns, Tm=20ns, 缺失需访问主存）
    Ta = 0.987 × 2 + 0.013 × (20 + 2) = 1.974 + 0.286 = 2.26ns
    （这里简化为 Ta = h×Tc + (1-h)×Tm = 0.987×2 + 0.013×20 = 2.234ns）

(4) 访问效率
    e = Tc / Ta = 2 / 2.234 = 89.5%
```

**例题2：** 256×256数组，行优先存储，Cache 8行×64B，直接映射

**程序A（行优先遍历 — 局部性好）：**
```
每行256个int = 1024B
每行需要 1024/64 = 16个数据块
每块第一次缺失（16个块中1个缺失）
命中率 = (16-1)/16 = 93.75%
```

**程序B（列优先遍历 — 局部性差）：**
```
每次访问a[i][j] → 跳256个int → 跨越1024B → 跨越16个数据块
每次访问都跳到不同的块 → 每次都不命中
命中率 ≈ 0%！
```

> **结论：** 写程序时注意遍历顺序！行优先访问性能远好于列优先

### Cache在现代CPU中的应用

```
Intel Pentium 4 (2.2GHz):
  L1 Cache: 指令和数据分离，各8KB，3个周期(1.5ns)
  L2 Cache: 256KB，8路组相联，20周期(10ns)
  L3 Cache: 512KB，8路组相联，30周期(15ns)
  主存: 16GB，400周期(200ns)
```

## 4.6 🌐 虚拟存储器

### 4.6.1 虚拟存储器原理

#### 为什么要虚拟内存？

> 🗣️ **比尔·盖茨(1981)：** "无论对谁来说，640K内存都足够了"
> 😅 现在：一个Chrome浏览器轻松吃掉1GB内存

**虚拟存储器 = 让程序以为它拥有一整片巨大的连续内存**

#### 核心思想

```
虚拟存储器是Cache思想在"主存-磁盘"层次的延伸！

Cache:     主存 → 缓存 → CPU        (加速)
虚拟内存:  磁盘 → 主存 → CPU        (扩大容量)
                  ↑
            按需加载，不是全部调入
```

### 4.6.2 页式虚拟存储器 ⭐

#### 基本概念

| 术语 | 英文 | 含义 |
|:----|:----|:------|
| **虚拟地址** | VA (Virtual Address) | 程序看到的地址（很大空间） |
| **物理地址** | PA (Physical Address) | 真正在内存条上的地址（通常较小） |
| **页** | Page | 数据和磁盘交换的单位（固定大小，通常4KB） |
| **页框** | Page Frame | 物理内存中存放一页的位置 |
| **页表** | Page Table | 记录虚拟页号→物理页号的映射 |
| **缺页** | Page Fault | 要访问的页不在主存中 → 从磁盘调入 |

#### 地址转换过程

```
虚拟地址 VA:
|←── 虚拟页号 VPN ──→|←── 页内偏移 VPO ──→|

步骤：
① CPU发出虚拟地址 VA
② MMU从VA中提取 VPN (虚拟页号)
③ 用VPN查页表 → 得到 PPN (物理页号)
④ PPN + VPO = PA (物理地址)
⑤ 用PA去访问Cache或主存
```

**页表结构：**
```
页表基址寄存器 PTBR → 页表基地址
                           +
VPN(虚拟页号) × 页表项大小 → 页表项PTE
                                  |
                          ┌────────┴────────┐
                          ↓                  ↓
                     有效位=1            有效位=0
                    PPN(物理页号)     → 缺页，磁盘中
```

### 缺页处理

```
页表项有效位 = 0 → 触发缺页异常 → 操作系统介入：

① 当前进程挂起
② 在磁盘中找到需要的页面
③ 如果主存满了 → 先淘汰一个页面（写入磁盘）
④ 从磁盘读入需要的页面到主存
⑤ 更新页表（有效位=1，填入PPN）
⑥ 重新执行触发缺页的指令

缺页损失 ≈ 10ms (磁盘操作) → 比Cache缺失大1000倍！
```

### 4.6.3 快表TLB ⭐

#### 为什么需要TLB？

```
查页表需要访问主存 → 每次内存访问都要先查一次页表
→ 一次数据访问变成两次内存访问 → 性能砍半！

解决：给页表也加一个Cache → TLB (Translation Lookaside Buffer)
```

#### TLB结构

```
TLB表项：(Valid, VPN, PPN)

CPU给出VPN → 在TLB中查找 → 命中 → 直接得到PPN → 不用访问页表！
                           → 缺失 → 去主存查页表 → 更新TLB
```

**整个地址转换流程：**
```
CPU → TLB命中？ → 是 → PPN → Cache命中？ → 是 → 数据
 ↓否                   ↓否                  ↓否
查页表                访存                  ↓
 ↓                    ← ← ← ← ← ← ← ← ← ← ↓
更新TLB              访问主存
```

### 虚拟存储器的管理方式对比

| 方式 | 分段单位 | 优点 | 缺点 |
|:----|:--------|:----|:-----|
| **页式** 📄 | 固定大小的页(4KB) | 无碎片，管理方便 | 页不是逻辑实体 |
| **段式** 📑 | 可变大小的段(函数/模块) | 符合程序逻辑，方便共享 | 有碎片，分配麻烦 |
| **段页式** 📚 | 段内分页 | 兼具两者优点 | 多次查表，速度慢 |

### 存储保护

> 多道程序运行时，防止程序A破坏程序B的数据

| 保护类型 | 说明 |
|:--------|:------|
| **地址越界** 🚧 | 访问的地址不在允许范围内 → 报错 |
| **访问越权** 🔒 | 试图写入只读区域 → 报错 |

**页表中的权限位：**
```
R/W: 可读可写（数据段）
R/O: 只读（常量数据）
R/E: 可读可执行（代码段）
```

## ⭐ 考试必背公式

| 公式 | 含义 |
|:----|:------|
| `h = Nc/(Nc+Nm)` | 命中率 |
| `Ta = h×Tc + (1-h)×Tm` | 平均访问时间 |
| `e = Tc/Ta` | 访问效率 |
| `Cache总容量 = 行数×(有效位+标记位+脏位+数据块大小)` | Cache硬件容量 |
| `交叉存取 t₁ = T + (n-1)τ` | 连续读n个字的交叉时间 |
| `顺序存取 t₂ = nT` | 连续读n个字的顺序时间 |
| `m = T/τ` | 交叉存取度 |
