## 9.1 🖱️ 输入输出设备概述

### I/O设备分类

| 类型 | 例子 | 速度 |
|:----|:-----|:----:|
| **输入设备** ⌨️ | 键盘、鼠标、扫描仪、麦克风、摄像头 | 慢~中 |
| **输出设备** 🖥️ | 显示器、打印机、音箱、投影仪 | 中~快 |
| **存储设备** 💾 | 硬盘、U盘、光盘、磁带 | 中~快 |
| **通信设备** 📡 | 网卡、调制解调器、蓝牙适配器 | 快 |

### I/O设备的三大特性

| 特性 | 含义 | 例子 |
|:----|:-----|:------|
| **异步性** ⏱️ | 设备速度远慢于CPU，何时准备好**不确定** | 键盘：你啥时候按键？不知道！|
| **实时性** ⚡ | 有些设备需要及时响应，不能等太久 | 打印机：缓存满了必须立刻处理 |
| **独立性** 🔄 | 各个设备独立工作，互不影响 | 键盘打字不影响鼠标移动 |

## 9.2 🔌 I/O接口

### 为什么需要I/O接口？

```
CPU和外部设备之间的"鸿沟"：
  ① 速度不匹配：CPU ns级 vs 设备 ms级
  ② 信号不匹配：CPU的数字信号 vs 设备的物理信号
  ③ 数据格式不匹配：CPU并行 vs 设备串行
  
→ 需要一个"中间人"来协调 → I/O接口
```

### I/O接口的功能

```
                          I/O接口
          ┌──────────────────────────────────────┐
          │  ┌────────┐ ┌────────┐ ┌──────────┐ │
   CPU ──→│  │地址译码 │ │数据缓冲 │ │格式转换   ││──→ 外设
  总线    │  └────────┘ └────────┘ └──────────┘ │
          │  ┌────────┐ ┌────────┐              │
          │  │状态/命  │ │中断处理 │              │
          │  │令寄存器│ └────────┘              │
          │  └────────┘                          │
          └──────────────────────────────────────┘
```

**I/O接口内部的关键组件：**
```
① 数据缓冲寄存器：暂存数据（缓解速度不匹配）
② 状态寄存器：记录设备状态（忙/空闲/错误）
③ 命令寄存器：接收CPU的指令
④ 地址译码：判断CPU要访问哪个设备
⑤ 控制逻辑：协调接口内部工作
```

### I/O接口编址方式 ⭐⭐

> **CPU怎么知道是在访问内存还是访问设备？**

#### ① 统一编址 (Memory-Mapped I/O)

```
把I/O设备的寄存器"映射"到内存地址空间中
访问设备 = 访问特定的内存地址

例：MIPS中
  LW $t0, 0xFFFF0000  ← 读键盘状态寄存器
  LW $t1, 0xFFFF0004  ← 读键盘数据寄存器

优点：
  ✅ 不需要专门的I/O指令（用LW/SW就行）
  ✅ 编程简单
  ✅ 所有访存指令都可用来控制设备

缺点：
  ❌ 占用内存地址空间（内存少了一块）
```

#### ② 独立编址 (Port-Mapped I/O)

```
I/O设备有独立的地址空间（与内存分开）
需要用专门的I/O指令访问

例：x86中
  IN AL, 60H    ← 从60H端口读键盘输入
  OUT 378H, AL  ← 往378H端口（并口）输出数据

优点：
  ✅ 不占用内存地址空间
  ✅ 有专门的I/O保护机制

缺点：
  ❌ 需要专用指令（IN/OUT）
  ❌ 编程相对复杂
```

| 对比 | 统一编址 🔗 | 独立编址 🚪 |
|:-----|:----------|:-----------|
| 指令 | 访存指令(LW/SW) | **专用I/O指令**(IN/OUT) |
| 地址空间 | 占用内存 | **独立**于内存 |
| 编程难度 | ✅ 简单 | ❌ 较复杂 |
| 保护机制 | ❌ 弱 | ✅ 强 |
| 典型架构 | MIPS, ARM, RISC-V | x86 |

## 9.4 🔄 程序查询方式

### 工作流程

> **"查一下→还没好→再查一下→还没好→...→好了！→处理"**

```
CPU启动设备
    │
    ▼
┌───────────────────┐
│ 读取设备状态寄存器 │ ←── CPU不断查询设备
└────────┬──────────┘
         │
    ┌────┴────┐
    │设备准备好了?│
    └────┬────┘
         │
    是 ┌──┴──┐ 否
       │     │
       ▼     └────→ 继续查询（循环）
   传输数据
       │
       ▼
   继续执行程序
```

**两种查询方式：**

| 方式 | 做法 | 优点 | 缺点 |
|:----|:-----|:-----|:------|
| **独占查询** 🔒 | CPU死等设备就绪 | 响应快 | **CPU完全浪费**在等待上 |
| **定时查询** ⏰ | CPU隔一段时间查一次 | CPU可以做别的事 | 响应可能有延迟 |

**实际例子：打印机查询**
```
CPU要打印"Hello"：
  ① 查打印机状态 → "忙"
  ② 查打印机状态 → "忙"
  ③ 查打印机状态 → "忙"
  ④ 查打印机状态 → "忙"
  ⑤ 查打印机状态 → "忙"
  ⑥ 查打印机状态 → "空闲！" → 发送一个字符'H'
  ⑦ 查打印机状态 → "忙"   ← 打印机在处理'H'
  ...

CPU 99%的时间都在"查状态"上！效率极低 ❌
```

## 9.5 ⚡ 程序中断方式 ⭐⭐

### 和程序查询方式的对比

```
程序查询方式：
  CPU多次主动查状态 → 浪费CPU时间

程序中断方式：
  设备告诉CPU"我准备好了" → CPU不用主动查

就像：
  查询 = 你每隔1分钟去问前台"快递到了吗？" → 累死了😫
  中断 = 前台打电话通知你"快递到了来取" → 轻松😊
```

**中断的优势：**
```
查询方式的时间线：
CPU ██████░░░░░░░░░░░░░░░░████████████████
     跑程序  一直查状态(浪费!)  传输数据

中断方式的时间线：
CPU ████████████████████████████│██████████
     跑程序(不受影响)           │处理完继续
                                  ↑
                            设备通知：准备好了！
```

### 中断处理流程 ⭐

```
主程序正在执行
    │
    ├── 取指令 → 执行指令
    │                        ← 此时有设备发中断请求
    ├── 取指令 → 执行指令
    │
    │     ┌─────────────────────────────────────
    │     │ 中断响应（硬件完成）
    │     │ ① 关中断（不再响应其他中断）
    │     │ ② 保存断点（将PC保存到EPC）
    │     │ ③ 中断识别（找到中断源）
    │     │
    │     │ 中断服务（软件完成）
    │     │ ④ 保护现场（寄存器入栈）
    │     │ ⑤ 开中断（允许更高优先级中断）
    │     │ ⑥ 中断服务（真正处理）
    │     │ ⑦ 关中断
    │     │ ⑧ 恢复现场（寄存器出栈）
    │     │ ⑨ 开中断
    │     │ ⑩ 中断返回（ERET→从EPC恢复PC）
    │     └─────────────────────────────────────
    │
    ├── 继续执行下一条指令
```

### 中断优先级和屏蔽 ⭐

#### 为什么需要优先级？

```
多个设备同时发中断 → CPU给谁服务？
→ 按优先级排队！

而且：高优先级的中断可以打断正在服务的低优先级中断
```

#### 响应优先级 vs 处理优先级

```
响应优先级 = 硬件决定的"默认顺序"
处理优先级 = 通过屏蔽字改变的"实际顺序"

通过设置中断屏蔽字：
  可以让原本优先级低的设备"暂时"获得高优先级处理
  就像：虽然官员级别高，但皇帝可以让他等一等
```

**中断屏蔽字示例：**
```
设备    响应优先级    屏蔽字(自己, 设备1, 设备2, 设备3)
设备1     最高          1    1    1    1   ← 屏蔽所有
设备2      高          0    1    0    0   ← 只让设备1中断自己
设备3      中          0    1    1    0   ← 让设备1、2中断自己
设备4      低          0    0    0    1   ← 只屏蔽自己
```

> ⚡ **DMA请求的优先级高于I/O中断请求！**
> 因为DMA对应高速设备，不及时处理会丢数据

**常见优先级顺序（从高到低）：**
```
硬件故障(最高)
    ↓
程序错误（除零、非法指令等）
    ↓
NMI（非屏蔽中断）
    ↓
DMA请求
    ↓
高速I/O设备
    ↓
低速I/O设备(最低)
```

## 9.6 🚀 DMA方式 ⭐⭐

### 为什么需要DMA？

> **中断方式每传1个字节，CPU要执行几十条指令（保护/恢复现场等）**
> **高速设备（如硬盘、网卡）每秒要传几百万次 → CPU忙不过来！**

**DMA = Direct Memory Access（直接存储器访问）**
**让硬件直接在外设和主存之间传数据，CPU不参与！**

### DMA控制器

```
                     CPU
                    ┌────┐
                    │    │←──── HOLD（总线请求）
                    │    │───→ HLDA（总线允许）
                    └────┘
                       │
     ┌─────────────────┼─────────────────┐
     │                 │                  │
  主存储器            DMA控制器           I/O设备
     │                 │                  │
     │  ←── 数据传 ──→ │  ←── 数据传 ──→ │
     │                 │                  │
                       │
             内部寄存器：
             ① 内存地址计数器（存到哪）
             ② 字节计数器（传多少）
             ③ 控制寄存器（方向等）
             ④ 状态寄存器（忙/空闲）
```

### DMA传输三阶段

```
第一阶段：预处理（CPU参与）
  ┌──────────────────────────────────┐
  │ CPU设置DMA控制器参数：             │
  │   • 数据方向（读/写）              │
  │   • 内存起始地址                   │
  │   • 传输字节数                     │
  │   • 设备编号                       │
  │ CPU→DMAC：初始化                  │
  └──────────────────────────────────┘
               ↓
第二阶段：传输（CPU不参与！）
  ┌──────────────────────────────────┐
  │ ① 设备准备好 → DREQ → DMAC       │
  │ ② DMAC → HOLD → CPU              │
  │ ③ CPU → HLDA → DMAC（交出总线）   │
  │ ④ DMAC控制总线，在设备和主存间传  │
  │ ⑤ 修改地址计数器、字节计数器       │
  │ ⑥ 传输完成 → 撤销HOLD → CPU收回   │
  │ 重复①~⑥直到所有数据传完           │
  └──────────────────────────────────┘
               ↓
第三阶段：结束（CPU参与）
  ┌──────────────────────────────────┐
  │ DMAC发中断 → CPU处理结束事宜       │
  │   • 检查传输是否成功               │
  │   • 处理剩余数据                   │
  └──────────────────────────────────┘
```

### 内存争用的三种处理方法

> **DMA传输时，DMA控制器和CPU都要访问内存 → 冲突！怎么办？**

| 方式 | 怎么做 | 优点 | 缺点 |
|:----|:------|:-----|:------|
| **停止CPU访存** 🛑 | DMA期间CPU暂停 | 控制简单 | CPU被浪费 |
| **周期挪用** ⏳ ⭐ | DMA挪用一个周期后归还 | **最常用**，效率高 | 控制稍复杂 |
| **交替访存** 🔄 | CPU和DMA轮流用一个周期的前半/后半 | 无需移交，最透明 | 硬件复杂 |

> **周期挪用的通俗理解：**
> 
> CPU和DMA共用一条"内存通道"
> 就像两人共用一扇门：
> - 停止CPU访存 = CPU让出门口，DMA一个人用（CPU站在旁边等）❌
> - 周期挪用 = DMA看到CPU不用门的时候"闪"进去，一个周期就出来 ✅

### DMA vs 中断 ⭐

| 对比项 | 中断方式 | DMA方式 |
|:-------|:--------|:--------|
| **数据传送方式** | **程序执行**（几十条指令） | **硬件控制** |
| **响应时机** | **指令之间** | **存储周期之间** |
| **CPU现场** | 需要保存和恢复 | **无需改变** |
| **处理能力** | 可以处理复杂事件 | **只能传数据** |
| **优先级** | 较低 | **高**（高速设备） |
| **适用设备** | **低速设备**（键盘、鼠标） | **高速设备**（硬盘、网卡） |

### DMA例题 ⭐

> **题目：** CPU主频500MHz，CPI=5，外设数据传输率5MB/s
> DMA块大小5000字节，DMA预处理+后处理需500时钟周期
> 求DMA方式下CPU的占用比例

**解答：**
```
① 传输5000字节的CPU开销
   CPU时间 = 500个时钟周期 / 500MHz = 500/500×10⁶ = 1μs

② 传输5000字节的总时间
   总时间 = 5000B / 5MB/s = 5000/5,000,000 = 1ms

③ CPU占用比例
   占用比例 = 1μs / 1ms = 0.1% ✅

结论：DMA方式几乎不占CPU时间！非常高效！
```

> **如果使用中断方式：**
> 每传输1个字节触发一次中断，每次中断几十条指令
> 传5000字节 → CPU占用率可能超过50%！😱

## 9.7 🏭 通道方式

### 为什么需要通道？

```
DMA的问题：
  ① 每个设备需要自己的DMA控制器 → 设备多了成本高
  ② 设备辅助操作（如磁盘寻道）仍需要CPU处理
  ③ DMA只能处理"传数据"，不能做设备管理

通道 = 专门的I/O处理器（IOP）
  - 可以执行通道程序（不仅仅是传数据）
  - 一个通道可以管理多个设备
  - 可以处理设备的各种辅助操作
```

### 通道分类

| 类型 | 传输单位 | 适用设备 | 工作方式 | 特点 |
|:----|:--------:|:--------:|:---------|:------|
| **字节多路通道** 🚶 | **字节** | 慢速设备（键盘、打印机） | 多个子通道轮流工作 | 效率高 |
| **选择通道** 🏃 | **数据块** | 高速设备（磁盘） | 一次只服务一个设备 | 独占，传完切换 |
| **数组多路通道** 🚗 | **数据组** | 中高速设备 | 交叉传输 | 介于两者之间 |

**通道与CPU的关系：**
```
CPU发出I/O请求 → 通道处理I/O → 完成时发中断 → CPU处理结果

CPU只管"安排任务" → 通道"自己干活" → 干完通知CPU
就像老板派活给项目经理 → 项目经理带团队干 → 干完汇报
```

## 🧠 四种控制方式终极对比 ⭐

| 方式 | 数据传输 | CPU角色 | 响应速度 | 适用设备 | 硬件复杂度 |
|:----|:---------|:--------|:--------:|:---------|:----------:|
| **程序查询** 🔄 | 程序控制 | 全程参与 | 快(主动查) | 低速 | **最简单** |
| **程序中断** ⚡ | 程序控制 | 服务程序 | 中(被通知) | 中低速 | 简单 |
| **DMA** 🚀 | **硬件控制** | **不参与** | 快(周期挪用) | **高速** | 较复杂 |
| **通道** ✈️ | 通道程序 | 基本不管 | 快(独立处理) | 高速批量 | **最复杂** |

## 🧠 本章口诀

```
I/O设备和CPU，速度相差千万倍
需要一个中间人，I/O接口来协调

数据传送四种法：
  程序查询最傻瓜，反复查看忙不忙
  中断方式省CPU，设备主动打报告
  DMA是直通车，硬件传数CPU歇
  通道更是高级货，I/O处理自己干

编址方式有两种：
  统一编址用访存，MIPS/ARM都在用
  独立编址IN/OUT，x86上面最常见

中断处理十步走：关→存→识→护→开→服→关→复→开→返
DMA传输三阶段：预处理→传数据→结尾处理
周期挪用效率高，高速设备全靠它

通道分类要分清：字节多路管慢速
选择通道管高速，数组多路折中好
```
